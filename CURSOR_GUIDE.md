# Working with Cursor on temflowral

Practical guidelines, not a style guide — Cursor already knows common conventions.
These are the things specific to this repo and this contract-first workflow.

## How Cursor rules work (current format)

Cursor no longer uses a single `.cursorrules` file. Rules live in `.cursor/rules/`
as `.mdc` files with frontmatter, version-controlled like any other file. Four
modes, controlled by frontmatter:

| Mode | Frontmatter | When it applies |
|---|---|---|
| Always Apply | `alwaysApply: true` | Every chat/agent session |
| Apply to Specific Files | `globs: <pattern>`, `alwaysApply: false` | Auto-attached when a matching file is in context |
| Apply Intelligently | `description: "..."`, `alwaysApply: false` | Agent reads the description and decides |
| Apply Manually | neither, `alwaysApply: false` | Only when you `@mention` the rule in chat |

A `.md` file in `.cursor/rules/` is ignored — it must be `.mdc`. This repo's rules
are checked in at `.cursor/rules/*.mdc` (see below); pull them into your local
checkout and they apply automatically.

## Rules for this repo

Five rule files, scoped by path so Go, TypeScript, YAML, and test code each get
relevant guidance without polluting each other's context:

- `00-project-overview.mdc` — always applied. Contract-first workflow, stack, team split.
- `api-contract.mdc` — `api/**/*.yaml`. Editing the OpenAPI spec.
- `backend-go.mdc` — `backend/**/*.go`. Go/Temporal conventions.
- `frontend-nextjs.mdc` — `frontend/**/*.{ts,tsx}`. Next.js/React Flow conventions.
- `playwright-testing.mdc` — test files. For the Playwright contributor.

Copy the `.cursor/rules/` folder from this delivery into the repo root and commit it.

## Day-to-day practices

**Review every diff before accepting.** Cursor's Agent mode can touch multiple
files in one pass — that's the point, but it means a bad instruction propagates
fast. Read the diff, don't just hit accept-all, especially for generated-code
regeneration (#6, #10 in `ISSUES.md`) where a stale flag can silently overwrite
hand-written code.

**Regenerate, don't hand-edit, generated code.** Once #6 (Go server interfaces)
and #10 (TS client) exist, anything under `backend/internal/generated/` or
`frontend/src/api/generated/` should be produced by the codegen command, not
edited directly by Cursor or by hand. Add a short comment header to those
directories (`// Code generated by oapi-codegen. DO NOT EDIT.`) so both you and
the agent respect the boundary — Cursor generally already avoids editing files
with that marker, but it isn't guaranteed, hence the rule file.

**Scope your chats.** Ask about the graph translator in one conversation and the
React Flow canvas in another rather than one long thread spanning both — shorter,
focused context produces better diffs than a chat that's accumulated the whole
day's work.

**Reference files, don't paste them.** `@openapi.yaml` or `@backend/internal/graph`
pulls the real, current file into context. Pasting a snapshot goes stale the
moment someone else edits it — with two people plus an automated-testing
contributor touching this repo, stale context is the likeliest source of bad
suggestions.

**Add a `.cursorignore`.** Keep generated/build output and anything sensitive out
of the indexed context:

```
backend/coverage.out
frontend/.next/
frontend/node_modules/
frontend/test-results/
*.env
*.env.local
```

**Never let Cursor commit secrets.** No `.env` values, Temporal connection
strings with credentials, or API keys in prompts or generated code — same rule
as any AI tool. `.cursorignore` above blocks indexing but doesn't stop you from
pasting a secret into chat manually, so treat it as a personal habit too.

**Use `/create-rule` when the agent repeats a mistake.** Don't pre-write rules
for hypothetical problems. If Cursor keeps doing something wrong (wrong import
style, editing generated files, wrong Temporal activity signature), correct it
once, then type `/create-rule` and let it capture the correction — that keeps
rules grounded in real patterns instead of speculative ones.

**Contract changes are a two-step review.** Because #2 (`openapi.yaml`) is the
thing both other tracks build against, treat any Cursor-generated change to it
as higher-stakes than a normal diff: check it against the CI spec-lint job (#5)
mentally before accepting, and flag breaking changes (renamed fields, removed
endpoints) in the devlog so the frontend and Playwright contributor aren't
surprised by a silent contract change.
